# CLAUDE.md

Anki Deck Agent — Cloudflare Workers + Telegram Bot that generates Anki flashcard decks via LLM + TTS.
Built with Deno, Hono, grammY. Infrastructure managed with OpenTofu.

## Commands

```bash
# Development
deno task dev:api            # API worker locally (denoflare)
deno task dev:processor      # Processor worker locally (denoflare)

# Quality (run before committing)
deno task check              # Type-check + lint + format check
deno task lint               # Lint only
deno task fmt                # Auto-format

# Testing
deno test                    # All tests
deno test --filter "name"    # Filter by test name
deno test workers/api/path_test.ts  # Single file

# Database
deno task db:generate        # Generate Drizzle migration
deno task db:migrate         # Apply migrations

# Infrastructure (from infra/environments/prod/)
tofu plan -var-file=../../terraform.tfvars
tofu apply -var-file=../../terraform.tfvars
```

## Architecture

```
workers/
├── api/        # Hono HTTP worker: Telegram webhook, user-facing API
├── processor/  # Queue consumer: LLM calls (via AI Gateway), TTS, card generation
└── shared/     # Shared types, D1 schema (Drizzle), R2 helpers, queue contracts
infra/
├── bootstrap/  # R2 bucket for OpenTofu state (one-time)
├── modules/    # Reusable: d1, r2, queues, tunnel
└── environments/prod/
```

Bindings: D1 (Drizzle ORM), R2 (assets), Queues (async tasks), AI Gateway (LLM proxy).
Local bindings (D1, R2, Queues) are synced via .env.infra generated by OpenTofu. Source this file before running
denoflare.

## Workflow Rules

### Execution Order

All workflow phases must follow this strict sequence:

1. **Memory Init** — Complete Initialization Hook (read all `.agent/` files) before any other action.
2. **Plan** — Discuss approach and confirm design with user.
3. **Test** — Write failing tests.
4. **Implement** — Write code to pass tests.

No phase may be skipped or reordered.

### Agentic Memory Protocol (.agent/)

You operate in a persistent workspace. State MUST be stored in `.agent/` files,
never only in chat context.

#### File Roles

| File           | Frequency  | Write Pattern | Purpose                       |
|----------------|------------|---------------|-------------------------------|
| `PLAN.md`      | Low        | Overwrite     | Architecture SSOT             |
| `SESSION.md`   | High       | Overwrite     | Current task state            |
| `DECISIONS.md` | Medium     | Append-only   | Architecture decision records |
| `BLOCKERS.md`  | On failure | Append-only   | Error patterns & resolutions  |

#### 1. Initialization Hook

AT THE START of every task:

1. Check if `.agent/` exists.

- If NOT: create the directory, create all four files with an `## EMPTY` stub,
  then ask the user to describe current project state before proceeding.
- If YES: read in order — `PLAN.md` → `DECISIONS.md` → `BLOCKERS.md` → `SESSION.md`.

2. Do NOT generate code or proposals until this read is complete.
3. Treat any missing or `## EMPTY` file as unknown state — ask, do not assume.

#### 2. Execution Hook

BEFORE modifying any code or IaC:

1. Rewrite `SESSION.md` with this exact schema:

```
   TASK: <one-line description>
   STATUS: IN_PROGRESS | BLOCKED | DONE
   STEPS:
     [ ] ...
     [x] ... (completed)
   LAST_ACTION: <what was just done>
   NEXT_ACTION: <what to do next if interrupted>
```

2. Update `SESSION.md` after every completed step.
3. On task completion: set STATUS=DONE, summarize in LAST_ACTION.

#### 3. Reflection Hook

On any test failure, build error, or API error:

- Attempt 1: Read the full error output → identify the specific cause → apply a targeted change → retry.
- Attempt 2 failure: STOP. Append to `BLOCKERS.md`:

```
  ### [YYYY-MM-DD] <Error Tag>
  CONTEXT: <file/command/binding>
  SYMPTOM: <exact error>
  ATTEMPTS:
    1. <tried>
    2. <tried>
  RCA: <root cause>
  RESOLUTION: <fix or UNRESOLVED>
```

- Before any operation on OpenTofu state, D1/R2 bindings, or Deno workspace config:
  search `BLOCKERS.md` for the relevant CONTEXT tag first.

#### 4. Decision Recording

When the user confirms or changes a design decision, append to `DECISIONS.md`:

```
### [YYYY-MM-DD] <Decision Title>
DECISION: <what>
RATIONALE: <why>
SUPERSEDES: <prior decision or N/A>
```

This file is append-only. Never modify existing entries.

#### 5. SSOT Maintenance

If architecture changes (directory structure, tech stack, service boundaries),
overwrite `PLAN.md` immediately. SESSION.md and DECISIONS.md take precedence over
chat context for historical facts.

### Plan → Test → Implement

> **Prerequisite**: This phase begins only after the Initialization Hook is complete.

1. **Plan first**: Discuss approach, trade-offs, and edge cases before writing code. Use `shift+tab` twice for plan mode
   on complex tasks.
2. **Write tests first**: Create test file (`*_test.ts`) with failing tests before implementation. Use `Deno.test()`
   with `describe`/`it` from `@std/testing/bdd`.
3. **Implement to pass**: Write minimal code to pass tests. Refactor after green.
4. **Verify**: Run `deno task check` before considering done. Fix all errors before committing.

### On Errors

When a test or type-check fails, do NOT blindly retry. Read the error, identify root cause, explain what went wrong,
then fix.

## Code Conventions

### TypeScript Strictness

- `noUncheckedIndexedAccess`, `noImplicitReturns`, `useUnknownInCatchVariables` enabled
- `eqeqeq`, `explicit-function-return-type`, `explicit-module-boundary-types`, `no-explicit-any`, `no-throw-literal`,
  `no-await-in-loop` enforced
- 120 char line width via `deno fmt`

### Patterns

- **Result types over exceptions**: Use discriminated unions `{ ok: true, data } | { ok: false, error }` for expected
  failures. Throw only for programmer errors.
- **Dependency injection**:
    - In `api/`: Pass bindings through Hono context (`c.env`).
    - In `processor/`: Pass bindings through the standard Queue handler arguments (`async queue(batch, env, ctx)`).
    - Never import bindings directly.
- **Explicit return types**: All exported functions and handler signatures must have explicit return type annotations.
- **Imports**: `jsr:` for Deno-native, `npm:` for Node/Cloudflare packages.
  See import maps in the root deno.jsonc. (Sub-workspaces like workers/api/deno.jsonc only declare package names).

### Cloudflare Workers Gotchas

- No global mutable state — isolates are reused across requests.
- Always `await` or `ctx.waitUntil()` promises. Floating promises = dropped work.
- D1 queries: use Drizzle ORM, never raw SQL in application code.
- Queue messages: payload must be < 128 KB (Cloudflare hard limit per message).
  Put large data in R2, pass R2 keys in queue messages.

### Testing

- Test files: `*_test.ts` co-located with source.
- Mock Cloudflare bindings in tests. Use Hono's `app.request()` for handler tests.
- For Drizzle/D1: test query-building logic with unit tests, not live D1.

### Commits

- Commitizen conventional commits (enforced by pre-commit hook).
- One logical change per commit.

### Design Constraints

**Dependency Rule (strict)**
All external dependencies must be injected, never instantiated at call site.
Cloudflare bindings received exclusively via `c.env` or queue handler `env`.

**Purity Boundary**
Business logic (validation, transformation, computation) = pure functions, no I/O.
Side effects (DB, queue, R2) permitted only in handler and repository layers.

**Layer Import Direction**
`shared/domain` → no external imports
`shared/ports` → imports domain only
`api/`, `processor/` → import shared, never each other

Violation of import direction is a hard error, not a style issue.

**Single Responsibility**
Handler: routing + serialization only.
Service: business logic only.
Repository: persistence only.
If a unit requires explanation of a second concern, decompose it.

**Interface Before Implementation**
Define the port (interface) in `shared/ports/` before writing any adapter.
Tests depend on the interface, never on the concrete implementation.
