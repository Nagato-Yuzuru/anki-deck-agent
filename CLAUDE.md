# CLAUDE.md

Anki Deck Agent — Cloudflare Workers + Telegram Bot that generates Anki flashcard decks via LLM + TTS. Built with Deno,
Hono, grammY. Infrastructure managed with OpenTofu.

## Build and Development

- **Priority 1 (Standard):** Always use `deno task <task_name>` defined in `deno.jsonc`.
- **Priority 2 (Deno Direct):** Use `deno run -A npm:<package>` for CLI tools.
- **Priority 3 (Fallback):** Use `pnpm dlx <package>` only if Deno's npm compatibility fails.
- **Strictly Prohibited:** Do not use `npm`, `npx`, or `yarn`. Avoid creating `package.json` unless absolutely
  necessary for external platform compatibility.

## Architecture

```
workers/
├── api/        # Hono HTTP worker: Telegram webhook, user-facing API
├── processor/  # Queue consumer: LLM calls (via AI Gateway), TTS, card generation
└── shared/     # Shared types, D1 schema (Drizzle), R2 helpers, queue contracts
infra/
├── bootstrap/  # R2 bucket for OpenTofu state (one-time)
├── modules/    # Reusable: d1, r2, queues, tunnel
└── environments/prod/
```

Bindings: D1 (Drizzle ORM), R2 (assets), Queues (async tasks), AI Gateway (LLM proxy). Local bindings (D1, R2, Queues)
are synced via `.env.infra` generated by OpenTofu. Source this file before running local dev.

## Workflow Rules

### Execution Order

All workflow phases must follow this strict sequence:

1. **Memory Init** — Read `.agent/` files (injected via hook). Do not proceed until complete.
2. **Plan** — Discuss approach and confirm design with user.
3. **Test** — Write failing tests.
4. **Implement** — Write code to pass tests.

No phase may be skipped or reordered.

### Plan → Test → Implement

1. **Plan first**: Discuss approach, trade-offs, and edge cases before writing code.
2. **Write tests first**: Create `*_test.ts` with failing tests. Use `Deno.test()` with `describe`/`it` from
   `@std/testing/bdd`.
3. **Implement to pass**: Write minimal code to pass tests. Refactor after green.
4. **Verify**: Run `deno task check` before considering done.

### On Errors

Do NOT blindly retry. Read the error, identify root cause, explain what went wrong, then fix.

## Code Conventions

### Patterns

**Result types with neverthrow**: Use `ResultAsync` / `Result` from `neverthrow` for all expected failures in the
service and domain layers. Do not use discriminated unions manually. Throw only for programmer errors (invariant
violations). Handler layer calls `.match()` to convert Result to HTTP response; unhandled throws are treated as 500.

**Dependency injection**:
- In `api/`: Pass bindings through Hono context (`c.env`).
- In `processor/`: Pass bindings through queue handler arguments (`async queue(batch, env, ctx)`).
- Never import bindings directly at call site.

**Pure ESM only**: No `require()`. All dependencies must have an ESM entry point. CJS-only packages require explicit
bundle compatibility verification before adoption.

### Hexagonal Architecture

Telegram / grammY is an **input adapter** (driven adapter). It must implement a port interface defined in
`shared/ports/`. Service layer must not import grammY directly — it operates on domain types only. This preserves the
ability to invoke the same service logic via other transports (HTTP, queue, CLI) without modification.

The same principle applies to all external integrations: AI Gateway, TTS APIs, and Telegram send APIs are output
adapters implementing ports defined in `shared/ports/`.

### Design Constraints

**Dependency Rule**: All external dependencies injected, never instantiated at call site. Cloudflare bindings received
exclusively via `c.env` or queue handler `env`.

**Purity Boundary**: Business logic (validation, transformation, computation) = pure functions, no I/O. Side effects
(DB, queue, R2, external API calls) permitted only in handler and repository/adapter layers.

**Layer Import Direction**:
```
shared/domain   → no external imports
shared/ports    → imports domain only
api/, processor/ → import shared, never each other
```
Violation of import direction is a hard error, not a style issue.

**Single Responsibility**: Handler: routing + serialization only. Service: business logic only. Repository: persistence
only. Decompose if a unit has two concerns.

**Interface Before Implementation**: Define the port in `shared/ports/` before writing any adapter. Tests depend on the
interface, never on the concrete implementation.

### Cloudflare Workers Gotchas

- No global mutable state — isolates are reused across requests.
- Always `await` or `ctx.waitUntil()` promises. Floating promises = dropped work.
- D1 queries: use Drizzle ORM, never raw SQL in application code.
- Queue messages: payload must be < 128 KB (hard limit). Put large data in R2, pass R2 key in queue message.
- Pure ESM runtime: no `require()`, no CJS-only dependencies.

### Observability

Use structured `console.log` / `console.warn` / `console.error` throughout. All log entries for significant events
must be JSON-serializable objects, not bare strings. This ensures Cloudflare Tail Workers and Logpush can parse and
filter entries.

```typescript
// Correct
console.log({ event: "deck_generated", userId, deckId, durationMs });
console.error({ event: "tts_failed", error: err.message, userId });

// Wrong
console.log("deck generated for " + userId);
```

Log levels: `log` for normal flow, `warn` for recoverable anomalies, `error` for failures that affect the user.

### Testing

- Mock Cloudflare bindings in tests.
- Use Hono's `app.request()` for handler tests.
- For Drizzle/D1: test query-building logic with unit tests, not live D1.
- Service layer tests operate on port interfaces (mocked adapters), never on concrete implementations.

### Commits

- Commitizen conventional commits.
- One logical change per commit.
